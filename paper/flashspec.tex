% ── Abstract ─────────────────────────────────────────────────────────────────
\begin{abstract}
Speculative decoding accelerates large language model (LLM) inference by
having a small \emph{draft} model propose $\gamma$ candidate tokens that a
larger \emph{target} model verifies in a single forward pass, preserving the
target model's output distribution exactly.
Existing systems have two practical gaps: the per-token accept/reject
decision is typically performed on the CPU---introducing a pipeline stall
proportional to vocabulary size---and the draft model is chosen offline,
ignoring shifts in prompt distribution across a deployment lifetime.

We present \flashspec, an open-source adaptive speculative-decoding engine
that addresses both.
Our first contribution is a Triton GPU kernel that performs the
accept/reject test entirely on-device, reading only two log-probability
scalars per candidate token.
The kernel's SRAM footprint is $O(1)$ in vocabulary size, independent of
whether the model has 32k or 128k tokens.
Our second contribution frames draft-model selection as a $K$-armed bandit
problem and solves it online with UCB1~\citep{auer2002finite} or Thompson
sampling~\citep{thompson1933,chapelle2011thompson}, adapting to non-stationary
acceptance rates without human retuning.

We prove that the output distribution of \flashspec{} is identical to
standard autoregressive sampling from the target model
(Theorem~\ref{thm:correctness}), and we enforce this guarantee continuously
in CI via a Kolmogorov--Smirnov test at significance $\alpha{=}0.01$
over 10{,}000 samples.

Initial measurements on a Tesla~T4 GPU with TinyLlama-1.1B-Chat
(4-bit NF4 quantization, $\gamma{=}4$) achieve
\textbf{44.2~tok/s} at a mean acceptance rate $\bar{\alpha} = 0.75$
and p50 step latency of 22.1~ms.
The UCB1 bandit achieves empirical cumulative regret of 100.2 over
$T{=}10{,}000$ rounds ($K{=}3$ arms), well within the $O(\sqrt{KT\log T})$
theoretical upper bound of ${\approx}526$.
Full benchmarks on Llama-3-8B/70B and comparison with
Medusa~\citep{cai2024medusa} and EAGLE~\citep{li2024eagle} on H100 SXM5
are in progress; all benchmark code and committed results are available
in the repository.

\flashspec{} is released under Apache~2.0 at
\url{https://github.com/Mattral/FlashSpec}.
\end{abstract}

% ── 1. Introduction ──────────────────────────────────────────────────────────
\section{Introduction}
\label{sec:intro}

Autoregressive generation in large language models is a memory-bandwidth-bound
sequential process: each token requires a full pass through the model,
loading all parameters from device memory.
At inference time this dominates end-to-end latency for both batch and
single-user deployments.

\emph{Speculative decoding}~\citep{leviathan2023fast,chen2023accelerating}
breaks this bottleneck by exploiting the asymmetry between the cost of
\emph{generating} tokens (one forward pass per token in the draft model)
and the cost of \emph{verifying} them (one forward pass for all $\gamma$
draft tokens simultaneously in the target model).
Under mild conditions on the acceptance rate, the expected number of tokens
generated per target forward pass is
$\E[\text{tokens}] = \gamma\bar{\alpha} + 1$,
where $\bar{\alpha}$ is the mean per-token acceptance probability.
At $\bar{\alpha} = 0.75$ and $\gamma = 4$, this is 4.0~tokens per step
versus 1.0 for vanilla autoregressive decoding.

Despite widespread adoption~\citep{cai2024medusa,li2024eagle}, two practical
gaps remain in open-source implementations:

\begin{enumerate}
  \item \textbf{CPU-side verification.}
  The per-token acceptance test (Equation~\ref{eq:accept}) is typically
  implemented in Python/PyTorch on the CPU, requiring a device-to-host
  synchronization at every decoding step.
  This creates a pipeline stall proportional to the vocabulary size $V$,
  which for modern LLMs ranges from 32k to 128k tokens.

  \item \textbf{Static draft selection.}
  The draft model is chosen once, offline, and held fixed throughout a
  deployment.
  In practice, acceptance rates vary substantially across prompt domains:
  a drafter tuned for conversational text may accept far fewer tokens on
  source code or long-context summarization, leaving throughput on the table.
\end{enumerate}

\flashspec{} addresses both gaps within a single, provably correct, and
extensively tested open-source library.

\paragraph{Contributions.}
\begin{itemize}
  \item A Triton GPU verification kernel with $O(1)$ SRAM usage in
        vocabulary size, eliminating the CPU synchronization bottleneck
        (Section~\ref{sec:kernel}).
  \item An online UCB1/Thompson bandit for adaptive draft-model selection
        with provable $O(\sqrt{KT\log T})$ regret, requiring no manual
        retuning (Section~\ref{sec:bandit}).
  \item A formal proof of output distribution equivalence to the target
        model for both the rejection sampling and typical-acceptance variants
        (Section~\ref{sec:correctness}).
  \item An open-source implementation with a full test suite,
        including a KS-based distribution-equivalence test (CI hard gate),
        property-based tests via Hypothesis, Triton kernel parity tests
        (float32 and bfloat16), and adversarial bandit chaos tests.
\end{itemize}

% ── 2. Background ────────────────────────────────────────────────────────────
\section{Background}
\label{sec:background}

\subsection{Speculative Decoding}

Let $p(\cdot \mid \mathbf{x})$ denote the target model's conditional
distribution over vocabulary $\mathcal{V}$ given context
$\mathbf{x} \in \mathcal{V}^*$, and $q(\cdot \mid \mathbf{x})$ the draft
model's distribution.
\citet{leviathan2023fast} showed that autoregressive sampling from $p$ can
be emulated exactly using the following procedure (Algorithm~\ref{alg:spec}):

\begin{enumerate}
  \item Draw $\gamma$ tokens $x_1, \ldots, x_\gamma$ autoregressively from $q$.
  \item Run one target forward pass to obtain
        $p(\cdot \mid \mathbf{x}_{<1}), \ldots, p(\cdot \mid \mathbf{x}_{<\gamma})$.
  \item For $i = 1, \ldots, \gamma$: accept $x_i$ with probability
        \begin{equation}
          \label{eq:accept}
          \alpha_i = \min\!\left(1,\;\frac{p(x_i \mid \mathbf{x}_{<i})}
                                          {q(x_i \mid \mathbf{x}_{<i})}\right).
        \end{equation}
  \item On rejection at position $i^*$, sample a residual token from
        \begin{equation}
          \label{eq:residual}
          p_{\text{res}}(x) \propto \max\!\left(0,\;
            p(x \mid \mathbf{x}_{<i^*}) - q(x \mid \mathbf{x}_{<i^*})\right).
        \end{equation}
\end{enumerate}

\begin{algorithm}[t]
\caption{Speculative Sampling (Leviathan et al., 2023)}
\label{alg:spec}
\begin{algorithmic}[1]
\REQUIRE context $\mathbf{x}$, draft model $q$, target model $p$,
         speculation length $\gamma$
\ENSURE next-token sequence sampled from $p$
\STATE Draft $\gamma$ tokens: $(x_1,\ldots,x_\gamma) \sim q(\cdot|\mathbf{x})$
\STATE Score: $p_i \leftarrow p(x_i|\mathbf{x}_{<i})$ for $i=1,\ldots,\gamma$
\STATE Draw $u_i \sim \mathrm{Uniform}(0,1)$ for each $i$
\FOR{$i = 1$ \TO $\gamma$}
  \IF{$u_i < \min(1,\; p_i / q_i)$}
    \STATE accept $x_i$
  \ELSE
    \STATE sample $x^* \sim p_{\mathrm{res}}$; \textbf{break}
  \ENDIF
\ENDFOR
\STATE \textbf{return} accepted tokens
\end{algorithmic}
\end{algorithm}

\subsection{Multi-Armed Bandits}

The K-armed bandit problem~\citep{robbins1952} models sequential
decision-making under uncertainty: at each round $t$, an agent selects
arm $k_t \in \{1,\ldots,K\}$ and observes reward $r_t \in [0,1]$.
Cumulative regret after $T$ rounds is
\begin{equation}
  R_T \defeq \sum_{t=1}^{T} \bigl(\mu^* - \mu_{k_t}\bigr),
\end{equation}
where $\mu^* = \max_k \mu_k$ is the optimal arm's expected reward.

\paragraph{UCB1}~\citep{auer2002finite} selects
\begin{equation}
  k_t = \arg\max_k \left[\hat{\mu}_k(t) + c\sqrt{\frac{2\ln t}{n_k(t)}}\right],
\end{equation}
where $\hat{\mu}_k(t)$ is the empirical mean reward of arm $k$ after $n_k(t)$
pulls.
UCB1 achieves $\E[R_T] \leq O(\sqrt{KT\ln T})$ regret.

\paragraph{Thompson Sampling}~\citep{thompson1933} maintains a
Beta$(\alpha_k, \beta_k)$ posterior for each arm and selects
$k_t = \arg\max_k \theta_k$ where $\theta_k \sim \mathrm{Beta}(\alpha_k,\beta_k)$.
It achieves $\E[R_T] \leq O(\sqrt{KT\ln T})$ with tighter empirical
constants~\citep{chapelle2011thompson}.

\subsection{System Overview}

Figure~\ref{fig:pipeline} shows the \flashspec{} generation loop.

\subsection{On-Device Verification Kernel}
\label{sec:kernel}

\paragraph{Motivation.}
The accept/reject test in Equation~\ref{eq:accept} is
embarrassingly parallel across the batch and $\gamma$ dimensions: each
position $(b, i)$ requires only the two scalars
$p(x_i^{(b)}\mid \cdot)$ and $q(x_i^{(b)}\mid \cdot)$---that is, the
log-probabilities at the \emph{selected draft token index}, not the full
vocabulary distribution.
Existing implementations nonetheless gather these values on the CPU,
paying a $\Theta(B\gamma)$ host-device transfer and one global synchronization
per step.

\paragraph{Kernel design.}
\texttt{verify\_tokens} is a Triton~\citep{triton2019} kernel that tiles over
the flat sequence of $B \times \gamma$ positions with a configurable
\texttt{BLOCK\_SIZE} $\in \{64, 128, 256, 512\}$, autotuned per GPU.
Each thread block:
\begin{enumerate}
  \item Gathers $\log p_i$ and $\log q_i$ from the respective vocabulary
        distributions using the draft token index.
        \emph{No other vocabulary logits are accessed.}
  \item Computes the log acceptance ratio:
        $\ell_i = \log p_i - \log q_i$ (computed in float32, upcast from
        bfloat16 inputs to avoid underflow).
  \item Accepts if $u_i < \min(1, e^{\ell_i})$, equivalently if
        $\log u_i < \ell_i$ (avoids the \texttt{exp}).
  \item Writes one boolean and contributes to the \texttt{first\_rejection}
        reduction per batch row.
\end{enumerate}

\paragraph{Complexity.}
SRAM usage per block is $O(\texttt{BLOCK\_SIZE})$---constant in vocabulary
size $V$.
Memory traffic per step is $O(B\gamma)$ reads (two scalars per position)
plus $O(B)$ writes, independent of $V$.
The pure-PyTorch reference implementation incurs $O(B\gamma V)$ reads when
gathering from the full logit tensor; the kernel reduces this by a factor of
$V$.

\paragraph{Numerical correctness.}
All acceptance comparisons use log-space arithmetic to prevent underflow.
The residual distribution (Equation~\ref{eq:residual}) is computed as
\begin{equation}
p_{\text{res}}(x) = \frac{\max(0,\, p(x) - q(x))}%
  {\sum_{x'}\max(0,\, p(x') - q(x')) + \varepsilon},\quad \varepsilon=10^{-9},
\end{equation}
with no additional temperature scaling applied to the residual.
Temperature is applied by the caller \emph{before} $\log\mathrm{softmax}$,
not after---a distinction that is easy to get wrong and difficult to detect
at default temperature $T=1$ (see Section~\ref{sec:lessons}).

\subsection{Online Bandit Draft Selection}
\label{sec:bandit}

\paragraph{Formulation.}
At each decoding step $t$, \flashspec{} selects one of $K$ registered draft
models.
The reward for pulling arm $k$ is the fraction of $\gamma$ draft tokens
accepted: $r_t = n_{\text{accepted}} / \gamma \in [0,1]$.
The bandit's goal is to minimise cumulative regret $R_T$ over $T$ steps.

\paragraph{UCB1 selector.}
Arm $k$ is assigned score
\begin{equation}
  \mathrm{UCB}(k, t) = \hat{\mu}_k + c\sqrt{\frac{2\ln t}{n_k}},
\end{equation}
with exploration constant $c=1$ and optional windowed statistics
(window size $W$, default $W=500$) to track non-stationary distributions.
Unpulled arms receive score $+\infty$ and are always explored first.

\paragraph{Thompson selector.}
Maintains $\mathrm{Beta}(\alpha_k, \beta_k)$ per arm, updated as
$\alpha_k \mathrel{+}= n_{\text{accepted}}$,
$\beta_k \mathrel{+}= \gamma - n_{\text{accepted}}$.
At each step, one sample is drawn per arm and the argmax is selected.

\paragraph{Thread safety and serialisation.}
Both selectors are thread-safe via a per-instance \texttt{threading.Lock}
and support JSON serialisation/deserialisation for checkpointing across
generation sessions.

\subsection{Correctness Guarantee}
\label{sec:correctness}

\begin{theorem}
\label{thm:correctness}
The joint distribution over the sequence of tokens accepted by
\flashspec{} (Algorithm~\ref{alg:spec} with either the UCB1 or Thompson
bandit selecting the draft model at each step) is identical to the joint
distribution produced by standard autoregressive sampling from the target
model $p$.
\end{theorem}

\begin{proof}[Proof sketch]
The correctness of the per-step distribution follows directly from
Theorem~1 of~\citet{leviathan2023fast}: for any draft model $q$ and any
context $\mathbf{x}$, the mixture of (i) accepted draft tokens with
probability $\alpha_i$ and (ii) residual tokens from $p_{\text{res}}$
produces marginal distribution equal to $p(\cdot \mid \mathbf{x})$ at every
accepted position.
The bandit's arm-selection policy at step $t$ is a function of the
\emph{history} of past acceptances only, not of the current draft tokens or
the target model's distribution at the current step.
Therefore, conditioning on any fixed arm selection $k_t$, the per-step
distribution is preserved by the Leviathan et al.\ argument.
By the tower property of conditional expectation, the joint distribution over
all accepted tokens is preserved regardless of the bandit's policy.
\end{proof}

The guarantee is continuously enforced in the test suite via a
Kolmogorov--Smirnov test (Section~\ref{sec:testing}).

% ── 4. Implementation ────────────────────────────────────────────────────────
\section{Implementation}
\label{sec:implementation}

\subsection{Software Architecture}

\flashspec{} is a typed Python package (Python $\geq 3.11$) organised as:

\begin{itemize}
  \item \textbf{\texttt{flashspec.kernels}}:
        Triton \texttt{verify\_tokens} and \texttt{gather\_accepted} kernels,
        with a pure-PyTorch reference (\texttt{\_reference}) for
        cross-platform use and parity testing.
        The package imports Triton inside a \texttt{try/except ImportError},
        so \texttt{pip install flashspec} succeeds on Windows and macOS
        (where Triton provides no official wheels)
        and raises a clear actionable error only when the kernel is
        \emph{called} on an unsupported platform.
  \item \textbf{\texttt{flashspec.sampling}}:
        \texttt{rejection\_sample} (Algorithm~\ref{alg:spec}) and
        \texttt{typical\_sample} variant.
  \item \textbf{\texttt{flashspec.bandit}}:
        \texttt{UCB1Selector}, \texttt{ThompsonSelector},
        \texttt{OracleSelector} (experiments only).
  \item \textbf{\texttt{flashspec.engine}}:
        \texttt{DraftModel} protocol with entry-point registry,
        \texttt{TargetModel} wrapper, \texttt{SpeculativeEngine}.
  \item \textbf{\texttt{flashspec.metrics}}:
        \texttt{AcceptanceTracker}, \texttt{ThroughputTracker}
        (with MFU), \texttt{LatencyTracker} (p50/p95/p99).
\end{itemize}

\subsection{Testing}
\label{sec:testing}

The test suite comprises four categories:

\begin{itemize}
  \item \textbf{Unit tests}: per-module tests with failure messages on every
        assertion. Hypothesis property-based tests cover all functions with
        integer shape parameters (shape invariant, range invariant,
        padding invariant).
  \item \textbf{Integration tests}: include the \emph{KS hard gate} — a
        Kolmogorov--Smirnov test at $\alpha=0.01$ over $N=10{,}000$ samples
        comparing the speculative sampler's output to the target distribution.
        A failing KS test blocks the CI build with no exceptions.
        ONNX export parity at \texttt{atol}=$10^{-5}$ for batch sizes 1--32.
  \item \textbf{Chaos tests}: adversarial bandit scenarios (0\% acceptance,
        100\% acceptance, sudden arm swap, $K=20$ arms).
  \item \textbf{GPU tests} (\texttt{@pytest.mark.gpu}):
        Triton kernel parity against the PyTorch reference in float32
        (\texttt{atol}=$10^{-5}$) and bfloat16 (\texttt{atol}=$10^{-3}$)
        at canonical shapes.
\end{itemize}

CI runs on every commit via GitHub Actions
(lint $\to$ test-cpu $\to$ coverage $\geq 95\%$ $\to$
onnx-parity $\to$ docs-build), with security scanning via
\texttt{pip-audit} and Trivy.

% ── 6. Engineering Lessons ───────────────────────────────────────────────────
\section{Engineering Lessons}
\label{sec:lessons}

We record four implementation lessons that may be useful to practitioners
building similar systems.

\paragraph{Temperature must be applied before \texttt{log\_softmax}.}
The correct computation is \texttt{log\_softmax(logits / T)}, not
\texttt{log\_softmax(logits) / T}.
These are numerically identical at $T=1$ (the default), making this bug
invisible in any test that does not exercise $T \neq 1$.
In the initial implementation, \texttt{rejection\_sample()} accepted a
\texttt{temperature} argument that had zero effect: the log-probabilities
it received were already computed by the target model's scorer without any
temperature scaling.
The fix required identifying the correct architectural layer for temperature
scaling (the verifier's \texttt{score\_draft()}) and removing the parameter
from the sampling function entirely.

\paragraph{The KS test sample size matters.}
The distribution-equivalence test was initially running at $N=1{,}000$
samples; the specification required $N=10{,}000$.
A KS test at $N=1{,}000$ has substantially lower power to detect subtle
distribution shifts than one at $N=10{,}000$.
The discrepancy was invisible at $N=1{,}000$ because no test in the suite
was failing; the gap was found only by systematic comparison of the code
against the specification document.
The test now runs at $N=10{,}000$ and is a hard CI gate.

\paragraph{Platform-specific dependencies break the install silently.}
The initial releases (v0.1.0--v0.1.2) listed \texttt{triton>=3.0.0} as
an unconditional core dependency.
Triton publishes official PyPI wheels for Linux only; the package was
uninstallable on Windows and macOS.
The fix moves Triton to a \texttt{[gpu]} optional extra with a
\texttt{platform\_system == 'Linux'} environment marker, and wraps the
Triton imports in a \texttt{try/except ImportError} with a clear
actionable error message on unsupported platforms.

\paragraph{Property-based tests find bugs that example tests miss.}
After adding Hypothesis property-based tests, the framework immediately
found an indexing bug in the rejection-sampling module at
$\gamma=1, B=1$ --- a shape that no hand-written test had exercised.
We recommend adding property-based tests for any function where integer
parameters affect output shapes or index ranges.

% ── 7. Related Work ──────────────────────────────────────────────────────────
\section{Related Work}
\label{sec:related}

\paragraph{Speculative decoding.}
\citet{leviathan2023fast} proved that autoregressive sampling from $p$ can
be emulated exactly using a cheaper draft $q$.
\citet{chen2023accelerating} independently proposed a parallel variant.
Both require manual selection of a static draft model.

\paragraph{Multi-head and feature-level drafting.}
Medusa~\citep{cai2024medusa} attaches $K$ independent decoding heads to
the target model, eliminating the separate draft model at the cost of
model surgery.
EAGLE~\citep{li2024eagle} conditions the drafter on the target model's
penultimate-layer features.
Neither adapts the draft strategy online.

\paragraph{Online learning for inference.}
To our knowledge, \flashspec{} is the first system to apply online bandit
algorithms to adaptive draft-model selection.
\citet{auer2002finite} established the UCB1 regret bound used here.
Thompson sampling~\citep{thompson1933} has been widely applied to
recommendation systems~\citep{chapelle2011thompson}; its application to
draft selection is novel.

\paragraph{GPU kernel optimisation for transformers.}
FlashAttention~\citep{dao2022flashattention} demonstrated that
IO-aware Triton kernels~\citep{triton2019} can match or exceed cuBLAS on
attention computation.
\flashspec{} applies the same principle---minimising HBM traffic by
operating on the minimum necessary data---to the verification step.

% ── 8. Conclusion ────────────────────────────────────────────────────────────
\section{Conclusion}
\label{sec:conclusion}

We presented \flashspec{}, an adaptive speculative-decoding engine that
addresses the CPU-side verification bottleneck and static draft-selection
limitation simultaneously.
The Triton verification kernel achieves $O(1)$ SRAM usage in vocabulary size
by reading only the two log-probability scalars relevant to each candidate
token.
The online bandit selector satisfies a formal $O(\sqrt{KT\ln T})$ regret
bound and converges empirically to well within the theoretical limit on the
first hardware experiment.

Initial measurements on a Tesla T4 with TinyLlama-1.1B-Chat (NF4, $\gamma=4$)
achieve 44.2~tok/s at $\bar{\alpha}=0.75$.
Full benchmarks on Llama-3-8B/70B-Instruct and Mistral-7B, and comparisons
with Medusa and EAGLE on H100~SXM5, are in progress and will be added to
the repository upon completion.

\flashspec{} is released under Apache~2.0.
The software, test suite, documentation, and all committed benchmark results
are at \url{https://github.com/Mattral/FlashSpec}.
A JOSS software paper is currently under review.
